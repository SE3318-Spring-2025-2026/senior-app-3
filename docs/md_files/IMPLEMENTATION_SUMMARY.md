### 2. ✅ Migration System - Reversible Database Versioning

**Files Created:**

#### a. Migration Runner - `backend/migrations/migrationRunner.js`
Core utility for managing migrations with:
- Idempotent migration execution (`runMigrationUp`)
- Reversible rollback support (`runMigrationDown`)
- Migration state tracking in MongoDB
- Status reporting and history
- Full up/down cycle support

**Key Functions:**
```javascript
getAppliedMigrations()        // List applied migrations
isMigrationApplied(name)      // Check if migration applied
recordMigration(name)         // Mark as applied
removeFromMigrationLog(name)  // Mark as rolled back
runMigrationUp(migration, db) // Apply migration
runMigrationDown(migration, db) // Rollback migration
getMigrationStatus(migrations) // Show applied/pending
resetAllMigrations(migrations, db) // Full reset
```

#### b. Migration 001 - `backend/migrations/001_create_user_schema.js`
**Purpose:** Documents baseline User schema structure  
**Status:** Idempotent (safe to run multiple times)
- Up: Verifies User collection exists; acts as documentation
- Down: Drops User collection (destructive, confirmation required)
- Maps all fields to OpenAPI UserAccount specification

#### c. Migration 002 - `backend/migrations/002_add_githubUsername_unique_constraint.js`
**Purpose:** Adds sparse unique index on githubUsername  
**Status:** Idempotent, handles index conflicts gracefully
- Up: Creates sparse unique index allowing multiple nulls
- Down: Removes the unique constraint
- Automatically drops any conflicting indexes before recreation

#### d. Migration Index - `backend/migrations/index.js`
Exports all migrations in order for sequential application:
```javascript
[001_create_user_schema, 002_add_githubUsername_unique_constraint]
```

#### e. Migration CLI - `backend/src/migrate.js`
Command-line interface for migration management:
```bash
npm run migrate:status  # Show applied/pending migrations
npm run migrate:up      # Apply all pending migrations  
npm run migrate:down    # Rollback last migration
npm run migrate:reset   # Reset all migrations (destructive)
```

**Package.json Updates:** [backend/package.json](./package.json)
```json
{
  "scripts": {
    "migrate:status": "node src/migrate.js status",
    "migrate:up": "node src/migrate.js up",
```

## Architecture Notes

**Design Decisions:**
1. **Mongoose + MongoDB** - Leverages existing backend setup; no new dependencies
2. **Sparse Unique Index** - Allows multiple unlinked users while preventing duplicate GitHub accounts
3. **Custom Migration System** - Simple, version-controlled, reversible; no external tools
4. **Idempotent Migrations** - Safe to run multiple times; handles existing state
5. **Comprehensive Documentation** - Single source of truth for schema, constraints, and workflows

**Performance Considerations:**
- Email index for fast login lookup (~O(1) for email-based queries)
- userId index for internal references in JWT tokens
- githubId index for OAuth provider mapping
- githubUsername index for GitHub integration lookups
- No cross-field indexes needed initially; add if query patterns show need

## Troubleshooting

### Issue: "Index already exists with different name"
**Solution:** Migration 002 handles this by dropping conflicting indexes first

### Issue: Multiple null values in sparse unique index
**Solution:** This is expected behavior. The sparse constraint correctly prevents:
- ✓ Multiple null values (expected for unlinked users)
- ✗ Multiple non-null duplicates (expected rejection)

### Issue: Migration fails to connect
**Check:**
1. MongoDB is running: `mongod` process or cloud connection
2. Connection string is correct in `.env`: `MONGODB_URI`
3. Database exists: `use senior-app`

### Issue: Tests timeout
**Solution:**
1. Ensure test MongoDB instance is running
2. Use `MONGODB_TEST_URI` env var for separate test database
3. Increase Jest timeout if needed: `jest.setTimeout(10000)`
