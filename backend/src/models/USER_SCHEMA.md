# User Account Schema (D1)

## Overview

The User schema defines the persisted user account record in the D1 User Accounts data store. The schema is designed to match the OpenAPI `UserAccount` specification ([docs/apispecs2.1.yaml](../docs/apispecs2.1.yaml), lines 48-78) while supporting additional implementation-specific fields for security, authentication, and profile linkage.

**Model File:** [src/models/User.js](./User.js)  
**Mongoose Version:** 7.0.0+  
**Database:** MongoDB  
**Collection Name:** `users`

---

## OpenAPI UserAccount Schema

The following fields are **required** to match the OpenAPI specification:

| Field | Type | Required | Unique | Indexed | Default | Description |
|-------|------|----------|--------|---------|---------|-------------|
| **userId** | String | ✓ | ✓ | ✓ | `usr_${uuid}` | System-generated unique identifier (e.g., `usr_8f3d2a`) |
| **email** | String | ✓ | ✓ | ✓ | — | User's email address; lowercased and trimmed; must be valid email format |
| **role** | String | ✓ | ✗ | ✗ | `student` | User's role; must be one of: `student`, `professor`, `admin`, `coordinator`, `system` |
| **githubUsername** | String | ✗ | ✓ (sparse) | ✓ | `null` | Linked GitHub username; linked during process 1.3 (OAuth completion); unique for non-null values |
| **emailVerified** | Boolean | ✓ | ✗ | ✗ | `false` | Has user's email been verified via verification token? |
| **accountStatus** | String | ✓ | ✗ | ✗ | `pending` | Account status; must be one of: `pending`, `active`, `suspended` |
| **createdAt** | Date | ✓ | ✗ | ✗ | `Date.now()` | Timestamp of account creation (ISO 8601); auto-generated by MongoDB timestamps |

### Field Constraints

#### userId
- **Type:** String
- **Generation:** UUID-based unique identifier formatted as `usr_${first-8-chars-of-uuid}` (e.g., `usr_8f3d2a`)
- **Constraints:** Required, unique, indexed for query performance
- **NULL:** Not allowed
- **Example:** `usr_8f3d2a`

#### email
- **Type:** String (email format)
- **Constraints:** Required, unique, indexed, lowercase, trimmed
- **NULL:** Not allowed
- **Validation:** Must match email regex; duplicate emails rejected
- **Case Sensitivity:** Normalized to lowercase on insert/update
- **Example:** `alice@university.edu`

#### role
- **Type:** String (enum)
- **Valid Values:** `student`, `professor`, `admin`, `coordinator`, `system`
- **Default:** `student`
- **Required:** Yes
- **NULL:** Not allowed
- **Description:** Defines user's role in the system; determines access control and features

#### githubUsername
- **Type:** String
- **Constraints:** Optional, unique (sparse), indexed, lowercase, trimmed
- **NULL:** Allowed (multiple nulls permitted)
- **Sparse Index:** Allows multiple users with null githubUsername; unique constraint applies only to non-null values
- **Linking:** Set during process 1.3 (GitHub OAuth connection); absent until user completes GitHub auth
- **Case Sensitivity:** Normalized to lowercase on insert/update
- **Example:** `alice-gh`

**Sparse Unique Index Behavior:**
```javascript
db.users.createIndex({ githubUsername: 1 }, { unique: true, sparse: true })
```
- ✓ User A: `{ githubUsername: null }` — OK
- ✓ User B: `{ githubUsername: null }` — OK (sparse allows multiple nulls)
- ✓ User C: `{ githubUsername: "alice-gh" }` — OK
- ✗ User D: `{ githubUsername: "alice-gh" }` — REJECTED (duplicate non-null value)

#### emailVerified
- **Type:** Boolean
- **Default:** `false`
- **Required:** Yes
- **NULL:** Not allowed
- **Description:** Flag indicating whether user's email has been verified via verification token
- **Transition:** `false` → `true` during email verification flow (process 1.4)

#### accountStatus
- **Type:** String (enum)
- **Valid Values:** `pending`, `active`, `suspended`
- **Default:** `pending`
- **Required:** Yes
- **NULL:** Not allowed
- **Transitions:**
  - `pending` → `active`: After email verification and onboarding completion (process 1.6)
  - `active` → `suspended`: By admin action (security or policy enforcement)
  - `suspended` → `active`: By admin action (reinstatement)
  - `pending` → (deleted): If verification fails; account may be removed

#### createdAt
- **Type:** Date (ISO 8601)
- **Default:** Generated by Mongoose `timestamps: true` option
- **Required:** Yes (auto-generated)
- **NULL:** Not allowed
- **Format:** ISO 8601 (e.g., `2025-04-02T14:30:00.000Z`)
- **Description:** Server-side timestamp of account creation; immutable after creation
- **Timezone:** UTC (stored in MongoDB as UTC, returned as ISO 8601)

## Implementation Fields

The following fields support authentication, security, and profile linkage but are **NOT** part of the OpenAPI spec:

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| **hashedPassword** | String | ✓ | — | Bcrypt hash of user's password (for local auth) |
| **githubId** | String | ✗ | `null` | GitHub's numeric user ID (OAuth identifier) |
| **studentId** | String | ✗ | `null` | Linked student ID (from StudentIdRegistry); validated during registration (process 1.1) |
| **lastLogin** | Date | ✗ | `null` | Timestamp of user's last successful login |
| **loginAttempts** | Number | ✓ | `0` | Failed login attempt counter; resets on successful login |
| **lockedUntil** | Date | ✗ | `null` | Timestamp until user's account is locked (after max failed attempts) |
| **emailVerificationToken** | String | ✗ | `null` | Temporary token for email verification flow; expires after configured duration |
| **emailVerificationTokenExpiry** | Date | ✗ | `null` | Expiry timestamp for emailVerificationToken |
| **passwordResetToken** | String | ✗ | `null` | Temporary token for password reset; generated by admin; expires after 15 minutes |
| **passwordResetTokenExpiry** | Date | ✗ | `null` | Expiry timestamp for passwordResetToken |
| **updatedAt** | Date | ✓ | `Date.now()` | Timestamp of last update; auto-generated by Mongoose timestamps |

## Indexes

The following indexes optimize query performance:

```javascript
userSchema.index({ email: 1 });                    // Primary lookup by email
userSchema.index({ userId: 1 });                   // Primary lookup by userId
userSchema.index({ githubId: 1 });                 // OAuth identity lookup
userSchema.index({ githubUsername: 1 });           // GitHub username lookup (unique constraint)
```

### Index Usage Patterns

- **email**: Fast login lookup; email uniqueness enforcement
- **userId**: Used in auth tokens and internal references
- **githubId**: OAuth provider identity mapping
- **githubUsername**: GitHub linkage verification; unique constraint for non-null values

## Unique Constraints

### Email Uniqueness
```javascript
email: {
  type: String,
  required: true,
  unique: true,
  lowercase: true,
  trim: true,
}
```
- **Enforcement:** MongoDB unique index on `email` field
- **Behavior:** Two users cannot have the same email address
- **Error on Duplicate:** `MongoServerError: E11000 duplicate key error collection: senior-app.users index: email_1`
- **Rejection:** Attempt to create/update user with duplicate email rejected with error code 11000

### GitHub Username Uniqueness (Sparse)
```javascript
githubUsername: {
  type: String,
  default: null,
  unique: true,
  sparse: true,
  lowercase: true,
  trim: true,
}
```
- **Enforcement:** MongoDB sparse unique index on `githubUsername` field
- **Behavior:** Unique constraint applies only to non-null values; multiple users may have `null` githubUsername
- **Use Case:** Prevents duplicate GitHub accounts while allowing unlinked users
- **Error on Duplicate:** `MongoServerError: E11000 duplicate key error collection: senior-app.users index: githubUsername_1_unique`

### User ID Uniqueness
```javascript
userId: {
  type: String,
  default: () => `usr_${uuidv4().split('-')[0]}`,
  unique: true,
  required: true,
}
```
- **Enforcement:** MongoDB unique index on `userId` field
- **Behavior:** System-generated during account creation; guaranteed unique
- **Format:** Short UUID prefix (e.g., `usr_8f3d2a`)

## OpenAPI Compliance

### UserAccount Response Schema

When returning a User document via the API, the following fields map to the OpenAPI `UserAccount` schema:

```yaml
UserAccount:
  type: object
  properties:
    userId:
      type: string
      description: System-generated unique identifier
      example: "usr_8f3d2a"
    email:
      type: string
      format: email
      example: "alice@university.edu"
    role:
      type: string
      enum: [student, professor, admin, coordinator, system]
    githubUsername:
      type: string
      description: Linked GitHub username
      example: "alice-gh"
    emailVerified:
      type: boolean
      default: false
    accountStatus:
      type: string
      enum: [pending, active, suspended]
      default: pending
    createdAt:
      type: string
      format: date-time
      example: "2025-04-02T14:30:00.000Z"
```

**API Response (POST /onboarding/complete):**
```json
{
  "userId": "usr_8f3d2a",
  "email": "alice@university.edu",
  "role": "student",
  "githubUsername": "alice-gh",
  "emailVerified": true,
  "accountStatus": "active",
  "createdAt": "2025-04-02T14:30:00.000Z"
}
```

**Private fields (excluded from API responses):** `hashedPassword`, `emailVerificationToken`, `passwordResetToken`, `loginAttempts`, `lockedUntil`, etc.

## Dataflow References

This schema is used in the following business processes:

1. **Process 1.1 — Validate Student ID:** Student registration; `studentId` validation
2. **Process 1.2 — Create Account:** Account creation with `hashedPassword`, `email`, initial `accountStatus: pending`
3. **Process 1.3 — Connect GitHub:** GitHub OAuth; populate `githubUsername`, `githubId`
4. **Process 1.4 — Verify Email:** Email verification; set `emailVerified: true`
5. **Process 1.5 — Connect GitHub (Prof):** Professor GitHub linkage (credentials stored separately)
6. **Process 1.6 — Onboard User & Store Account:** Finalize onboarding; set `accountStatus: active`
7. **Group Creation:** Display user profile; read `userId`, `role`, `githubUsername`
8. **Committee Assignment:** Admin access; read/modify `role`, `accountStatus`
